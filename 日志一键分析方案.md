# 日志一键分析方案（基于现有 ADK 能力，按 SQL 参数约束优化）

## 1. 目标

在现有 `root_agent + route_by_skill + 日志分析 tools` 基础上，按你给定 SQL 口径实现一键链路：

1. 输入自然语言：`xx用户，xx时间发生xx问题`。
2. 强制提取 SQL 参数：`dt(年月日)`、`user_id`、`app_id(20=iOS, 21=Android)`。
3. 执行用户信息 SQL 查询。
4. 基于用户信息 + 时间 + 问题描述检索日志，并输出分析报告。

## 2. 复用现有能力

当前项目已经具备以下可直接复用能力：

1. `filter_logs`：日志筛选（时间、类型、级别、关键词等）。
2. `analyze_log_with_source`：日志异常模式统计 + 源码关联。
3. `analyze_and_generate_report`：CRISP-L 报告生成与落盘。
4. `route_by_skill`：确定性技能路由，适合接入新的一键编排 skill。
5. `root_agent` 编排能力：适合承载“自然语言一键分析”的入口。
6. `$start-live-flow-assistant`：开播链路 flow 聚合 + 源码关联 + CRISP-L 报告。

结论：日志分析主链路无需重写，只需要补齐“自然语言参数解析 + SQL 用户查询 + 一键编排”三个缺口，并在开播问题场景复用 `$start-live-flow-assistant`。

## 3. 新增最小能力设计

### 3.1 新增工具 1：`parse_incident_text`

作用：将自然语言输入解析成 SQL 可用的结构化参数。

输入：
- `incident_text: str`，例如：`user_id=123456，2026-04-17 iOS 发生开播失败问题`

输出（建议）：
- `dt`: `YYYY-MM-DD`（SQL 分区日期）
- `user_id`: 用户 ID（字符串）
- `app_id`: `20` 或 `21`
- `app_name`: `ios` 或 `android`（仅展示）
- `start_ts_ms`: 起始时间（毫秒，可选，用于日志筛选）
- `end_ts_ms`: 结束时间（毫秒，可选，用于日志筛选）
- `problem_desc`: 问题描述
- `keywords_hint`: 问题关键词建议（可选）
- `parse_confidence`: 解析置信度

解析规则：
1. 若文本出现 `iOS/ios/苹果`，映射 `app_id=20`。
2. 若文本出现 `Android/android/安卓`，映射 `app_id=21`。
3. 若已明确给 `app_id`，以数字为准，并校验只能是 `20/21`。
4. 若只给了时间段（例如 `2026-04-17 10:00~12:00`），`dt` 默认取开始时间所在日期，即 `2026-04-17`。

失败策略：
1. `dt`、`user_id`、`app_id` 任一缺失时，返回结构化错误，不进入 SQL。
2. `app_id` 非 `20/21` 时直接报错并提示可选值。

### 3.2 新增工具 2：`query_user_profile_by_sql`

作用：按 `dt + user_id + app_id` 查询用户信息。

输入：
- `dt`
- `user_id`
- `app_id`

SQL（按你的口径，参数化执行）：

```sql
select dt, app_id, platform, device_id, user_id, model, app_version, os_version
from gzlc_real.fact_wuxian_api_event_v2
where dt = :dt
  and user_id = :user_id
  and app_id = :app_id
  and LENGTH(device_id) > 0
limit 1
```

输出（建议）：
- `dt`
- `app_id`
- `platform`
- `device_id`（输出时脱敏）
- `user_id`
- `model`
- `app_version`
- `os_version`

实现建议：
1. 必须使用参数化 SQL（禁止字符串拼接），避免注入风险。
2. 查询结果为空时返回统一错误码 `USER_PROFILE_NOT_FOUND`。
3. 输出时对 `device_id` 做脱敏（保留前后少量字符）。

### 3.3 新增工具 3：`analyze_incident_one_click`

作用：编排“一键分析”全流程，内部复用现有日志工具。

编排步骤：
1. `parse_incident_text(incident_text)`
2. `query_user_profile_by_sql(dt, user_id, app_id)`
3. 基于 `problem_desc + user_profile` 生成日志检索参数：
   - 时间窗：`start_ts_ms/end_ts_ms`（若无明确时分秒，可用整天）
   - 关键词：`problem keywords + user_id + device_id + app_version + os_version`
   - 可选：`log_type`、`c_startswith`
4. `filter_logs(...)` 快速确认命中规模
5. `analyze_log_with_source(...)` 生成结构化分析
6. 路由分流：
   - 开播问题：调用 `route_by_skill(skill_name="start-live-flow-assistant", ...)`
   - 非开播问题：调用 `analyze_and_generate_report(...)`

返回结果建议：
- `parsed_incident`
- `sql_params`（`dt/user_id/app_id`）
- `user_profile`
- `selected_skill`（`start-live-flow-assistant` 或 `default-log-report`）
- `filter_summary`
- `analysis_summary`
- `report_path`
- `report_preview`

### 3.4 新增能力 4：复用 `$start-live-flow-assistant` 开播专项分析

触发条件（满足任一即可）：
1. `problem_desc` 包含 `开播`、`startLive`、`live flow`、`flowId`。
2. 用户在输入中显式指定 `$start-live-flow-assistant`。

执行方式：
1. 通过 `route_by_skill` 调用 `start-live-flow-assistant`。
2. 默认参数建议：
   - `c_startswith="1"`
   - `keywords="CSP_BIZ_WATCHCAR_STARTLIVE,flowId"`
   - `exclude_last_stage="recover_check_start"`
3. 时间窗继续复用解析后的 `start_ts_ms/end_ts_ms`。

输出增强：
1. 在统一返回中增加 `selected_skill="start-live-flow-assistant"`。
2. 除 `report_path` 外，补充 `json_path`（开播 flow 结构化结果）。
3. 报告标题建议默认为 `startLive 开播链路日志报告`。

### 4.3 prompt 指令扩展

在 `prompt.py` 里补充 root 编排规则：
1. 当用户输入符合“xx用户，xx时间发生xx问题”时，优先调用 `analyze_incident_one_click`。
2. 参数缺失时，优先提示补全 `dt/user_id/app_id`。
3. 明确说明 `app_id` 映射：`20=iOS，21=Android`。
4. 当 `problem_desc` 命中开播语义时，优先分流到 `$start-live-flow-assistant`。

## 6. 风险与兜底

1. `dt/user_id/app_id` 缺失：直接阻断并返回缺失字段列表。
2. SQL 空结果：返回“用户画像未命中”，并建议核对 `dt` 与 `app_id`。
3. 日志命中为 0：自动放宽一次条件（扩大时间窗或降级关键词）。
4. 开播专项未提取到有效 `flowId`：降级到通用 `analyze_and_generate_report` 路径，并标记降级原因。
5. 证据不足：报告中标记“条件化结论”，避免强结论。
6. 敏感数据：延续现有脱敏策略，确保 `device_id/token` 不明文输出。

## 7. 分阶段落地建议

1. 第一阶段：支持固定参数模板（`dt + user_id + app_id + 问题描述`）+ 开播语义分流到 `$start-live-flow-assistant`。
2. 第二阶段：支持自然语言 `iOS/Android` 自动映射 `app_id`，并支持“昨天/今天”等日期表达。
3. 第三阶段：引入用户维度日志索引（`user_id/device_id/app_version`）优化检索命中率。
