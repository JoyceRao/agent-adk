# 日志一键分析方案（更新版）

## 1. 目标

保持用户输入参数提取逻辑不变，在一键分析链路中强制执行以下步骤：

1. 从自然语言中提取 `dt / user_id / app_id`（保持现有规则，不做变更）。
2. 使用 `query_user_profile_by_sql(dt, user_id, app_id)` 查询用户信息。
3. 使用用户信息中的 `device_id + app_version`，调用 `apm_log_sql_assistant` 查询日志文件记录，拿到 `log_file_name`。
4. 拼接下载 URL：`download_url = '${LOG_DOWNLOAD_URL}?name=${log_file_name}'`。
5. 调用 `download_url_assistant(download_url)` 下载日志到本地。
6. 基于本地日志进行“xx用户，xx时间发生xx问题”的筛选、分析、报告输出。

## 2. 工具职责与顺序

### 2.1 `parse_incident_text`（保持不变）

输入：
- `incident_text`

输出（保持现状）：
- `dt`
- `user_id`
- `app_id`（20=iOS, 21=Android）
- `start_ts_ms`
- `end_ts_ms`
- `problem_desc`
- `keywords_hint`

失败规则（保持现状）：
- 缺失 `dt/user_id/app_id` 直接返回错误，不进入 SQL 查询。

### 2.2 `query_user_profile_by_sql`

输入：
- `dt`
- `user_id`
- `app_id`

输出：
- `user_profile`（重点字段：`device_id`、`app_version`、`os_version`）

失败规则：
- 查询失败或无数据时，终止链路并返回 `USER_PROFILE_NOT_FOUND` 或对应错误码。

### 2.3 `apm_log_sql_assistant`

输入：
- `dt`
- `app_id`
- `device_id`（来自用户画像）
- `app_version`（来自用户画像）

输出：
- `log_record.log_file_name`

失败规则：
- 无记录时返回 `LOG_RECORD_NOT_FOUND`，终止链路。

### 2.4 `download_url_assistant`

输入：
- `url = '${LOG_DOWNLOAD_URL}?name=${log_file_name}'`

输出：
- `saved_path`（本地日志文件路径）
- `filename`
- `size_bytes`

失败规则：
- 下载失败时终止链路，返回 `LOG_DOWNLOAD_FAILED` 或具体下载错误码。

### 2.5 本地日志分析工具

对 `saved_path` 执行：

1. `filter_logs`（先精确关键词，再放宽一次）。
2. `analyze_log_with_source`（日志 + 源码关联分析）。
3. `analyze_and_generate_report`（生成 CRISP-L 报告）。

说明：
- 分析对象必须是下载后的本地日志文件，不再依赖传入的原始日志路径。

## 3. 一键编排（`analyze_incident_one_click`）

### 3.1 编排顺序（必须）

1. `parse_incident_text(incident_text)`
2. `query_user_profile_by_sql(dt, user_id, app_id)`
3. `apm_log_sql_assistant(dt, app_id, device_id, app_version)`
4. 构建 `download_url = '${LOG_DOWNLOAD_URL}?name=${log_file_name}'`
5. `download_url_assistant(download_url)`
6. `filter_logs(saved_path, ...)`
7. `analyze_log_with_source(saved_path, ...)`
8. `analyze_and_generate_report(saved_path, ...)`

### 3.2 关键保证

1. 参数提取逻辑不变：`dt/user_id/app_id` 的识别和校验规则沿用原实现。
2. SQL 顺序固定：先用户画像，再日志文件记录。
3. 下载后再分析：所有筛选和报告都基于本地下载文件。
4. `log_path` 仅作为兼容参数保留，不再作为主分析输入。

## 5. 错误与兜底

1. 参数缺失：返回 `MISSING_REQUIRED_FIELDS`，提示补齐 `dt/user_id/app_id`。
2. 用户画像缺失：返回 `USER_PROFILE_NOT_FOUND`。
3. 日志文件记录缺失：返回 `LOG_RECORD_NOT_FOUND`。
4. 下载失败：返回 `LOG_DOWNLOAD_FAILED`（或下载工具具体错误码）。
5. 本地筛选无命中：放宽一次关键词；仍无命中则输出无数据报告。
